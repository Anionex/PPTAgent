1 管线概览

pptagent 的目标是：给定一个 Markdown 文稿 + 一份 PPTX 模板，产出与模板风格一致的结构化幻灯片。整个流程分两大阶段，每阶段内部又有若干子步骤。

Stage I · 模板引导（离线一次）

解析 PPTX → 功能页分类 → 内容页聚类 → 抽取每个布局的 content_schema。产物是 slide_induction.json，可反复复用。

Stage II · 内容生成（每次运行）

解析 Markdown → 生成 outline → 每页选布局 → LLM 产生 commands → eval Python 代码编辑 slide → autofit → 保存 PPTX。

2 Stage I · 模板引导

模板引导由 pptagent/induct.py::SlideInducter 执行。它把一份 PPTX 里的 样例幻灯片 拆成若干可复用的 布局，并让 LLM 读每张图，抽取"填空骨架" content_schema。

20 → 1920 张样例压缩为 19 个可复用布局
10 / 3 / 4文本布局 / 图文布局 / 功能页
1schema_extractor Agent 一次出 19 份 schema
2.1 category_split — 分离功能页

让 LLM 扫描所有 slide 的文字，找出哪些是 opening / table of contents / section outline / ending 这 4 类"功能页"。功能页不参与后续聚类。

产出： functional_cluster = {"opening": [1], "table of contents": [2], "section outline": [3, 7, 11, 15], "ending": [20]}（brand 模板 20 页中的 7 页被识别为功能页）
00_slide_induction_full.json（查看完整 layout_induction） prompts/category_split.txt
2.2 layout_split — 内容页聚类

先按 (layout_name, content_type) 做粗分，再用 ViT 视觉 embedding 做相似度聚类。每个聚类推一个 template_id（取形状最多的那一张）作为该布局的"代表"。

Fork 改动 · no GPU fallback

原实现要求 PyTorch + ViT 模型，否则崩溃。改后：image_model 返回空列表时，回退为 一张 slide → 一个布局，本报告就是在 macOS 无 GPU 环境下跑的。

brand 实测：20 张幻灯片被聚类成 15 个内容布局 + 4 个功能布局 = 19 个 layout key。
2.3 content_induct — 每个布局抽 schema（SlideSchema.response_model）

对每个 template_id 对应的代表 slide，把 HTML 喂给 schema_extractor（Agent LLM），让它识别 slide 上有哪些"元素槽"。

真实 content_schema 片段（opening 布局）

Element: decorative background text
    type: text
    suggested_characters: 8
    example: ['BUSINESS']   ← fork 新增 example 字段
    default_quantity: 1

Element: main title
    type: text
    suggested_characters: 12
    example: ['科技新品上市宣传介绍——']
    default_quantity: 1

Element: placeholder paragraphs
    type: text
    suggested_characters: 66
    example: ['Add your text content.Add your text content.Add your text content.', ...]
    default_quantity: 2
Fork 改动 · example 注入 · ContextVar 竞态修复
  • example 字段：原版 schema 只有参数名和字数，editor 常把"章节"写成 "ONE"。加入 example: [模板原始内容] 让 editor 看到"原本长什么样"。
  • ContextVar 竞态：原版用 ContextVar 存"当前 slide 的文本池"，asyncio.gather 并发时最后设置的值覆盖前面。改为 SlideSchema.response_model(text_fields, image_fields) 返回闭包捕获的 Pydantic 子类，每个 slide 独立。

3 Stage II · 内容生成

PPTAgent.generate_pres(source_doc, num_slides, ...) 编排整条链路：文档解析 → 生成 outline → 对每页执行 5 小步。

5 / 13请求 5 张内容页，功能页注入后变 13 页
4 种自动注入 opening · toc · section_outline × 5 · ending
max_at_once=2并发度限制（防 LLM 限流）
3.1 Markdown → Document（含 Summary）

Document.from_markdown() 调用解析模型（gemini-3-flash-preview）把 Markdown 切段，为每个 section 生成 summary，提取图片 caption。

输入 7.7 KB Markdown · ~190 行
输出 5 个 section · 每个带 summary · 含 6 张图 caption

3.2 generate_outline — Planner Agent 规划每页

planner 读文档 overview，输出 N 个 OutlineItem(purpose, topic, indexes, images)。indexes 指定这页用哪些 section 的哪些 subsection。

brand 实测 outline 第 4 项： 
{
  "purpose": "Introduce the 70-year journey of AI development, covering the
              origins at the Dartmouth Conference and early foundational milestones...",
  "topic": "Origins of Artificial Intelligence",
  "indexes": [{
    "section": "人工智能发展史",
    "subsections": ["人工智能七十年征程", "起源：达特茅斯会议", "早期奠基性里程碑"]
  }],
  "images": ["images/eniac.jpg", "images/ai_timeline.png"]
}
然后 _add_functional_layouts() 按规则注入功能页：
  1. 位置 0 插入 opening
  2. 位置 0 插入 table of contents
  3. 每遇到新 topic，在其前面插 section outline（本次注入 5 次）
  4. 末尾附加 ending

5 个内容页 + 1 opening + 1 toc + 5 section_outline + 1 ending = 13 页。

04_outline_generated.json（planner 原始输出） 06_outline_with_functional.json（注入后）
3.3 generate_slide — 单页 5 小步链路（概览）
  1. retrieve：outline_item.retrieve(slide_idx, source_doc) 从 Document 里拉出这页对应的 header（本页目标）、content_source（指定 subsection 原文拼接）、images（含 caption）。
  2. _select_layout：layout_selector LLM 读 header + content 类型，返回某个 layout 标题。
  3. _generate_content：content_organizer 按 layout 的 content_schema 生成 "command_list"：每个元素槽的 old_data → new_data 映射。
  4. _edit_slide：editor LLM 把 command_list 翻成 Python 代码块（调 replace_paragraph / del_image / clone_paragraph / ...）。然后 CodeExecutor.execute_actions() 用 eval() 逐行跑。
  5. autofit + save：统一做 _autofit_text() 防溢出，auto_size=NONE 锁形状大小，写 .pptx。
这条 5 步链路会重试 如果 eval() 报错、或 content_schema 校验不过，整条链路重试（最多 3 次），CodeExecutor 记录失败上下文喂给下一次的 LLM。

4 每页 5 步链路 · 真实追踪

选一页查看完整 retrieve → select_layout → generate_content → edit_slide 的真实输入输出。功能页（opening/toc/section_outline/ending）的 retrieve 数据由模板固定内容填充，不走 planner 的 indexes。

5 API 工具箱 · editor 能调的 5 个函数

editor LLM 的代码只能调 pptagent/apis.py::API_TYPES.Agent 注册的 5 个函数。每个函数失败时抛 SlideEditError，错误会喂回下一次重试。

函数作用典型调用（来自 brand 追踪）
replace_paragraph(div_id, para_id, text)替换某文本框某段文字replace_paragraph(7, 0, "人工智能发展简史")
del_paragraph(div_id, para_id)删段（删到剩一段时只清空）del_paragraph(5, 2)
clone_paragraph(div_id, para_id)克隆段（为填更多条目）clone_paragraph(5, 0)
replace_image(img_id, path)替换图片（含尺寸缩放）replace_image(2, "images/eniac.jpg")
del_image(figure_id)删图片del_image(0)
Fork 改动 · eval 鲁棒性
  • 中文引号修复：LLM 偶尔输出 "…"（U+201C/201D），eval() 直接报 SyntaxError。捕获后自动转义嵌套引号重试。
  • 空/损坏图片：replace_image 先验证文件存在且非空，给明确报错而不是 PIL 崩溃。
  • 幂等删除：remove_item 遇到不存在元素静默跳过，不抛 ValueError。
  • 装饰性元素：editor prompt 明确允许 new_data = []，不再强行编造内容。

6 Fork 主要改动（相对上游）

PPTX 解析补全（7 fix）
  • 多 slide master 全扫描（原来只扫第一个 → 0 slides）
  • 空布局名自动命名 layout_N
  • 嵌套 group shape 递归仿射变换 _remap_descendants()
  • fill.build() 后写坐标（避免被 spPr 重建覆盖）
  • add_picture() 显式回写 flipH/flipV
环境兼容（5 fix）
  • Docker 不可用 → warning（原来 sys.exit(1)）
  • 无 PyTorch/ViT → 每 slide 独立一布局
  • API 代理返回 markdown 包裹的 JSON → 剥壳 + Pydantic 验证
  • 代理不强制 schema → _validate_with_coercion()
  • Claude API MIME 校验 → 按实际文件类型设置
文本溢出 · 布局保真
  • 读取 <a:lnSpc><a:spcPct> 真实行距，而非硬编码 1.3
  • 非 CJK 字符宽度 0.55→0.50，避免英文行数高估
  • autofit 后设 auto_size=NONE 锁形状大小
  • 从 layout/master 直接删 >4 字符的装饰 TEXT_BOX
CLI 与架构
  • --template/-t 直通 PPTAgent.generate_pres() Python API
  • 绕过 MCP agent loop，避免 LLM 中介引入错误
  • 明确两条管线边界：deeppresenter（HTML）vs pptagent（模板）
